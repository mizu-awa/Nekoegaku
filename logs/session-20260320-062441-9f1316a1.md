## `extractContours` の `refTopY` は適切か？

現状のロジックは：

1. 上から走査して最初の不透明ピクセル → `topOuter`（線の外端＝上端）
2. そこから下に走査して最初の透明ピクセル → `topInner`（線の内端＝下端）
3. `refTopY = (topOuter + topInner) / 2` → **線の中央**

これは **正しい** と思います。

理由：`getTopY` で計算される曲線も「猫の輪郭線の中心パス」を表現しています。描画時に `ctx.lineWidth` で太さを付与するので、線の中心を基準にするのが整合的です。

もし `topOuter`（線の上端＝外端）だけを基準にすると、それは「黒線の外周」になり、線の太さ分だけズレます。`(topOuter + topInner) / 2` で線の中央をとっているのは、描画パスとの比較として適切です。

**ご懸念の「外周に相当するところをとっている」** という点についてですが、現状は `topOuter` ではなく `(topOuter + topInner) / 2` を使っているので、外周ではなく中央を取っています。問題ないはずです。

ただし、もし実際の描画結果を見て「正解線とプレイヤーの線がズレている」ように見える場合は、別の原因（lineWidthの違い、画像のアンチエイリアシング、`alphaThreshold` の値など）が考えられます。何か具体的にズレが見えていますか？

## 問題の構造

`refBottomBaseline` の計算式（[game.js:415](js/game.js#L415)）:

```js
refBottomBaseline[idx] = refBottomY[idx] - getFeetContribution(xr, answer, W, H)
```

つまり「画像から検出した足の波形」から「`answerParams` の数式による足の波形」を引き算して、ベースラインを逆算しています。

**しかし、画像は手描きの有機的な波形であるのに対し、`getFeetContribution` は `sin(4.5 * 2π * t)` という完全な正弦波です。** この2つが完璧に一致しないと、引き算の残差がベースラインにノイズとして残ります。

## 確認方法

ブラウザのコンソールで以下を実行すると、画像から検出した生の `refBottomY`（ベースライン算出前）が正しいか確認できます:

```js
// 足領域のrefBottomYを可視化
const W = 800, H = 400;
const startPx = Math.floor(0.031 * W);
for (let i = 0; i < 20; i++) {
  const px = Math.floor(0.165 * W) + i * 10;

  console.log(`px=${px}, refBottomY=${refBottomY[Math.round(idx)]?.toFixed(1)}`);

```

- `refBottomY` 自体が滑らかな波形 → 画像認識は正常。`answerParams` と画像のミスマッチが原因
- `refBottomY` がガタガタ → 画像認識側の問題（スケーリングのアンチエイリアスや `alphaThreshold` の影響）

**おそらく前者**（answerParamsの正弦波と手描き波形のズレ）だと思います。手描きの波は周期・振幅・位相が場所ごとに微妙に違うので、単一の `sin` では完全に打ち消せません。
